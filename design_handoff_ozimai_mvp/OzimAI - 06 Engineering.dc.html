01 · Архитектура

Архитектурный стиль и обзор системы

Модульный монолит, не микросервисы. Команда 3–5 инженеров, скорость важнее изоляции; модули (Диалоги, Расписание, CRM, Знания, Биллинг) — отдельные пакеты с чёткими границами внутри одного деплоя, готовые к вынесению в сервисы при росте (>1 200 клиентов, см. 01 Executive). AI-инференс и очередь сообщений — единственные компоненты, масштабируемые отдельно с день-1.

WhatsApp/Instagram ⇄ Channel Gateway → очередь (события) → Orchestrator (диалог, состояние, function calling) ⇄ LLM Router (провайдер A/B)
Orchestrator ⇄ Модули: Знания (RAG) · Расписание (слоты, блокировки) · CRM (контакты, статусы) · Отчёты
API (REST + WS) ← Панель (SPA) · Билинг-модуль ⇄ Kaspi Pay / карты · Job-планировщик → напоминания, отчёты, реактивация

Ключевое решение: Orchestrator — единая точка, где происходит function calling (запись/перенос/отмена), поэтому гарантии консистентности (атомарная блокировка слота) реализуются в одном месте, а не размазаны по каналам.

02 · Стек

Технологический стек

FrontendReact + TypeScript SPA, mobile-first (документ 03/04). Real-time через WebSocket для живой ленты диалогов.
BackendNode.js (TypeScript), NestJS-модули = границы монолита. Очередь событий — Postgres-based (напр. pg-boss) на MVP, миграция на Kafka/SQS при масштабе.
ДанныеPostgreSQL (основные данные, RLS для мультиарендности) · Redis (кэш, блокировки слотов, сессии) · Vector store (pgvector) для RAG базы знаний.
AILLM Router с абстракцией провайдера (примary + fallback) — снижает риск ценовой/доступностной зависимости (риск из 01 Executive). Function calling для действий; guardrails-слой перед и после генерации (запрет медсоветов, проверка цен по базе).
ИнфраструктураОблако с регионом в КЗ/СНГ (или EU + локальный DPA до появления локального дата-центра уровня прод); контейнеры + managed Postgres; IaC (Terraform); CI/CD на каждый merge в main.
НаблюдаемостьСтруктурные логи + трейсинг диалогов (каждый AI-ответ — с промптом, версией базы знаний, латентностью); алерты на: канал недоступен, латентность >5с p95, доля эскалаций аномально высока.
03 · Данные

Схема данных (основные сущности)

Сущность Ключевые поля и связи
organizationid, name, plan, locale, billing_status → 1:N branches, users
branch / practitionerpractitioner: name, org_id, branch_id, working_hours (jsonb) → 1:N slots
serviceorg_id, name, price, duration_min, version — версионирование прайса (edge case из 02 Product)
patient (contact)org_id, phone (уникален в рамках org), name, lead_status, source_channel → 1:N conversations, appointments
conversation / messageconversation: patient_id, channel, status (ai/escalated/human); message: sender_type, text, ai_meta (jsonb: model, latency, escalation_reason)
appointmentpatient_id, practitioner_id, service_id, slot_start, status (booked/confirmed/no_show/cancelled), created_by (ai/human)
knowledge_docorg_id, source_type, content, embedding (vector), version, published_at
audit_logorg_id, actor (ai/user_id), action, entity, before/after (jsonb) — покрывает NFR аудита из 02 Product

Мультиарендность: org_id на каждой таблице + Postgres Row-Level Security — политика по умолчанию deny, явные policy на чтение/запись в рамках своей организации. Одна база на MVP; шардирование по org_id — при необходимости после Y1.

06 · Безопасность

Безопасность и соответствие

Данные и соответствие
  • Хранение перс. данных пациентов — регион КЗ/СНГ (закон РК «О персональных данных», решение 01 Executive)
  • Шифрование at-rest (AES-256) и in-transit (TLS 1.2+)
  • DPA (Data Processing Agreement) с каждой клиникой при онбординге
  • Право на удаление данных пациента по запросу; retention 90 дней после закрытия аккаунта
Доступ и приложение
  • RBAC: owner / admin / staff — на уровне организации
  • RLS в Postgres — изоляция между организациями на уровне БД, не только приложения
  • Секреты — vault-сервис, ротация ключей провайдеров
  • Guardrails AI: жёсткий системный промпт + пост-фильтр ответов (запрет медсоветов/цен вне базы) — тестируется в CI на наборе adversarial-примеров
