# Handoff: OzimAI MVP — AI-администратор для стоматологических клиник

## Overview

OzimAI — AI-сотрудник для частных медицинских и стоматологических клиник Казахстана. Он отвечает пациентам в WhatsApp/Instagram 24/7, записывает на приём, шлёт напоминания, ведёт мини-CRM и присылает владельцу еженедельный отчёт. Задача этого хендоффа — реализовать MVP как настоящий продакшн-продукт: backend, база данных, реальная интеграция с WhatsApp и LLM, и панель управления для администратора клиники.

Весь стратегический и продуктовый контекст (рынок, ниша, финмодель, риски, роадмап, ADR) зафиксирован в 8 документах, перечисленных в разделе Files — прочитай их перед началом работы, особенно 02 (Product/PRD), 06 (Engineering) и 07 (Development/ADR).

## About the Design Files

Файлы `OzimAI - *.dc.html` — это **дизайн-референсы**, написанные как HTML-прототипы (документы + один интерактивный прототип панели). Это не продакшн-код и не то, что нужно копировать напрямую. Задача — **реализовать этот дизайн и эту логику в реальном стеке**, описанном в документе 06 Engineering: Node.js/NestJS backend, PostgreSQL, React+TypeScript frontend. Прототип панели (`OzimAI - Prototype.dc.html`) — это работающая спецификация поведения (включая реальный вызов LLM с function calling для записи и guardrails) — пересоздай эту же логику на реальном бэкенде, а не как клиентский код.

## Fidelity

**Поведенчески референсный прототип** (`OzimAI - Prototype.dc.html`) — работающая модель того, как должна вести себя система (диалоги, эскалация AI↔человек, запись, календарь, тест-чат AI с function calling). Это не UI-финал для мобильных/пограничных случаев — детали такого рода уточняй по `OzimAI - 03 UX.dc.html` (error/empty/loading состояния, responsive-брейкпоинты).

## Стек и архитектура (обязательны к соблюдению — см. 06 Engineering и ADR в 07 Development)

- **Backend**: Node.js + TypeScript, NestJS-модули как границы модульного монолита (не микросервисы на старте — ADR-03). Модули: Диалоги/Orchestrator, Расписание, CRM, Знания (RAG), Биллинг.
- **База данных**: PostgreSQL с Row-Level Security для мультиарендности (`org_id` на каждой таблице) + pgvector для базы знаний. Redis — кэш, блокировки слотов, очередь.
- **Frontend**: React + TypeScript SPA, mobile-first, WebSocket для живой ленты диалогов.
- **AI**: LLM Router с primary+fallback провайдером (абстракция, не хардкод одного вендора). Guardrails — системный промпт + пост-фильтр, **AI никогда не даёт медицинских консультаций** (ADR-02, юридически критично для ниши).
- **Хранение данных**: регион КЗ/СНГ обязательно (закон РК «О персональных данных» + врачебная тайна — ADR-04). Это архитектурное решение, не деталь на потом.
- **Интеграции**: WhatsApp Business Platform через официального BSP-партнёра (не серый API), Kaspi Pay/карты для биллинга.

Полная схема данных, API-контракты и диаграмма архитектуры — в `OzimAI - 06 Engineering.dc.html`, разделы 01–04.

## MVP-скоуп (обязательно — не добавляй лишнего)

Из `OzimAI - 02 Product.dc.html`, раздел MoSCoW — **Must**, только это в первом релизе:
1. AI-диалог в WhatsApp (рус/каз), ответ ≤5 сек, только из базы знаний клиники, эскалация при неуверенности
2. Запись/перенос/отмена в календаре, атомарная блокировка слота
3. Напоминания за 24ч/2ч с кнопками подтвердить/перенести/отменить
4. Мини-CRM (автозаполнение из диалогов)
5. Панель: лента диалогов, перехват одной кнопкой, календарь, база знаний, отчёты
6. Онбординг-мастер (загрузка прайса → готовый AI ≤1 день)
7. Биллинг: триал 14 дней, самостоятельная оплата, read-only при неуплате

**Явно вне скоупа (Won't)**: AI-телефония, мобильное приложение, полноценная BI-аналитика, интеграции МИС/YClients (только CSV-экспорт), маркетинговые рассылки, любые медицинские консультации от AI.

Полный инвентарь экранов, вайрфреймы и потоки — `OzimAI - 03 UX.dc.html`. Кратко:

### 1. Диалоги (главный экран)

- **Layout**: 3 колонки — список диалогов (300px) | окно диалога (flex, min 400px) | карточка пациента (260px). При узких экранах — горизонтальный скролл, не сжатие (см. "Files" для рабочего примера этого фикса).

- **Статус-чипы**: ⚑ Ждёт человека (warn), ✓ Записан (accent), AI ответил (нейтральный), Вы отвечаете (info), печатает… (faint) — см. Design Tokens за точные цвета
- **Диалог**: сообщения пациента слева (белые), AI справа (зелёный accent-050 фон), эскалация — жёлтый warn-фон вместо зелёного, админ — тёмный ink-фон. Каждое AI-сообщение подписано "✦ Айым"

### 2. Календарь
- Список слотов по врачу (чипы-переключатели врачей), клик по свободному — создать запись, по занятому — отменить (в проде это делает AI/админ, не прямой клик — см. примечание в прототипе)

### 3. Пациенты (мини-CRM)
- Таблица: имя, статус, заметка, кнопка "Открыть диалог" (переход в контекст диалога)

### 4. Айым (AI и база знаний)
- Слева — список пар "вопрос → ответ" базы знаний
- Справа — тест-чат: реальный вызов LLM с системным промптом (см. Design Tokens/AI Behavior ниже) и function calling (check_availability, book_appointment, escalate)

### 5. Отчёты
- 3 KPI-карточки (деньги от AI, записей AI, неявки), переключатель недель

### 6. Онбординг-мастер (4 шага, см. 03 UX)
- Шаг 1: клиника (название, адрес, часы, врачи+графики)
- Шаг 2: знания (загрузка прайса файлом/фото/текстом, парсинг с предпросмотром "услуга→цена")
- Шаг 3: канал (подключение WhatsApp по QR, имя/тон AI)
- Шаг 4: тест-чат — владелец сам пробует "сломать" AI перед запуском

## Interactions & Behavior

- **Эскалация**: при медицинском вопросе, вопросе вне базы знаний, или агрессии/2 циклах спама — AI вызывает `escalate`, статус диалога → "attention", администратор получает уведомление
- **Запись**: AI вызывает `check_availability` → предлагает ≤3 слота → после согласия вызывает `book_appointment` с атомарной блокировкой слота (TTL 5 мин на hold, см. API `/appointments/:id/hold` в 06 Engineering)
- **Двойная запись**: при конкурентной попытке — второй пациент получает соседние слоты, не ошибку
- **Печатает-индикатор**: 3 анимированные точки во время реального ожидания ответа LLM
- **Автонапоминания**: джоб-планировщик, за 24ч и 2ч до визита
- **Guardrail-тест**: живой пример в прототипе — задать вопрос "что выпить от боли" → AI обязан эскалировать, не советовать лекарство
- **Markdown в ответах AI**: LLM пишет обычный текст без разметки (см. системный промпт) — если разметка всё же проскакивает, зачищай `**`/`*`/backtick перед рендером

## State Management

Ключевые сущности (полная схема — 06 Engineering §03):
- `organization`, `branch`/`practitioner`, `service` (с версионированием прайса)
- `patient` (contact), `conversation`/`message` (с `ai_meta`: модель, латентность, причина эскалации)
- `appointment` (статусы: booked/confirmed/no_show/cancelled)
- `knowledge_doc` (с embedding для RAG, версионирование)
- `audit_log` (обязателен для каждого действия AI — actor, action, before/after)

Мультиарендность — `org_id` + RLS на каждой таблице, политика по умолчанию deny.

## AI Behavior (system prompt reference)

Рабочий системный промпт из прототипа (адаптируй под реальную базу знаний клиники, подгружаемую динамически по `org_id`):

```
Ты — Айым, AI-администратор стоматологии «Айгуль» в Алматы. Отвечай коротко,
тепло и по-человечески, максимум 2-3 предложения. Пиши на языке пациента
(русский или казахский).

БАЗА ЗНАНИЙ (единственный источник фактов и цен): [подставляется из knowledge_doc]

ЖЁСТКИЕ ПРАВИЛА:
1. Никогда не давай медицинских советов, не называй лекарства, не ставь
   диагнозы. При любом медицинском вопросе вызови tool escalate.
2. Называй только цены и факты из базы знаний. Если чего-то там нет —
   вызови escalate, не придумывай.
3. Если пациент хочет записаться — сначала вызови check_availability,
   предложи 1-2 свободных времени, затем book_appointment.
4. Никогда не подтверждай запись, не вызвав book_appointment.
5. Пиши обычным текстом без markdown.
```

Tools (function calling): `check_availability({doctor?})`, `book_appointment({doctor, time, patientName})`, `escalate({reason})` — в проде реализуются как реальные вызовы к модулям Расписание/CRM через Orchestrator, не как клиентские функции.
